
# VIDEO AGENT FLAG — JETMYMOTO VIDEO PIPELINE OPERATING MANUAL

Use this section as the default operating guide for all JetMyMoto video tasks. Do not re-scan the full repository unless the task explicitly requires it. Start from `update_22.md`, `functions/video-engine/`, `scripts/e2e-render-pipeline.mjs`, and the active mission dataset source.

## Purpose

This manual is the source of truth for the current cinematic mission video system. The pipeline is intentionally split so Mapbox rendering happens once, while video variants can be regenerated repeatedly without re-triggering route or frame acquisition.

## Core architecture

```text
Mission slug
  -> mission lookup
  -> route geometry resolution
  -> buildFramePlan.ts
  -> Step 1: download/render/persist frames
  -> Step 2: FFmpeg composition from stored frames
  -> upload final videos
```

## Two-step contract

### Step 1 owns

- route resolution
- camera plan
- map frame generation
- baked map imagery
- route glow/core visual layer
- frame persistence
- metadata / manifest / completion marker

### Step 2 owns

- grading
- glow polish
- vignette
- grain
- blur
- CTA / hook text
- logos / watermark
- audio
- output aspect ratios
- variant regeneration

### Non-negotiable rule

Step 2 must never call Mapbox.

## Canonical operating rules

1. Never re-explore the whole repo unless required.
2. Never mix Step 1 and Step 2 responsibilities.
3. Never use Step 2 to fetch route geometry.
4. Never use file count alone as completion proof.
5. Never regenerate map frames for Step 2-only variant work.

Use `manifest.json` and `complete.json` as the completion contract.

## Storage contract

Step 1 output:

```text
mapboxrawframes/{slug}/
  frames/
    frame_0000.png or .webp
    ...
  metadata.json
  camera_state.json
  manifest.json
  complete.json
  archive/frames.tar.gz   (optional)
```

Step 2 output:

```text
ffmpeg1/{slug}/
  {slug}.mp4
  {slug}_9x16.mp4
  {slug}_1x1.mp4
```

Step 1 is complete only if all expected frames exist and `metadata.json`, `camera_state.json`, `manifest.json`, and `complete.json` are present. `complete.json` is the final completion signal.

## Mission lookup rule

Use slug as the primary key:

```text
slug
  -> mission dataset lookup
  -> if roadCoordinates exist, use them
  -> else buildRouteGeo.ts
  -> buildFramePlan.ts
```

Prefer stored mission coordinates over rebuilding route geometry unless rerouting is explicitly required.

## Frame cache rule

Treat the existing rendered frame library as a reusable mission frame cache. Before any new Step 1 job, check whether a valid frame set already exists for the slug by verifying:

- `complete.json` exists
- manifest is valid
- all expected frames are present
- format is recognized
- no zero-byte frames

If valid, skip Step 1 and go directly to Step 2.

## Step 1 guide

File:

```text
functions/video-engine/step1_downloadFrames.ts
```

Responsibilities:

- load mission
- resolve route geometry
- create frame plan
- render/download frames
- save frames locally
- upload frames to storage
- generate metadata artifacts
- write completion marker last

Step 1 must produce frames plus `metadata.json`, `camera_state.json`, `manifest.json`, and `complete.json`.

Step 1 must emit structured JSON logs with fields like:

```json
{
  "step": "step1",
  "slug": "mxp-to-muc-alpine-traverse",
  "event": "frame_download_start"
}
```

## Step 2 guide

File:

```text
functions/video-engine/step2_buildVideo.ts
```

Responsibilities:

- validate Step 1 completion
- sync frames locally if missing
- detect frame format from manifest
- resolve variant config
- generate filtergraph
- run FFmpeg
- upload outputs

Startup sequence:

1. load `complete.json`
2. load `manifest.json`
3. load `metadata.json`
4. load `camera_state.json`
5. sync missing frames locally
6. run strict manifest validation
7. generate FFmpeg outputs

Step 2 must never do route fetch, Mapbox calls, mission rerouting, or Step 1 completion guessing. It must support `16:9`, `9:16`, and `1:1`, with aspect-specific text layout. Use NVENC if available, otherwise fall back to `libx264`.

## Variant system rule

All variant behavior must flow through:

```text
functions/video-engine/utils/variantSchema.ts
```

If the task is to make a new video variant, modify only variant config and Step 2 invocation unless the map imagery itself must change.

## Manifest and camera rule

- `functions/video-engine/utils/storageManifest.ts` is the source of truth for storage paths, manifest generation, and strict validation.
- `functions/video-engine/utils/buildFramePlan.ts` is the canonical source for camera motion.

Do not duplicate manifest logic or camera math elsewhere.

# Wave 9 — MCP Stack Expansion: Mapbox + FFmpeg Direct Pipeline

- Timestamp: `2026-04-06`
- Scope: Eliminated Remotion dependency for map/video generation by wiring Mapbox and FFmpeg as direct MCP servers.

## MCP Server Stack (`.vscode/mcp.json`)

| Server | Package | Purpose |
|--------|---------|---------|
| `remotion-video-engine` | `@remotion/mcp@latest` | Legacy Remotion composition rendering (retained for fallback) |
| `mapbox` | `@mapbox/mcp-server@0.11.0` | Direct Mapbox API access — replaces Remotion's Mapbox GL dependency |
| `ffmpeg` | `ffmpeg-mcp-server` | Video manipulation, filter graphs, metadata burn-in |

## Mapbox MCP Server — 28 Tools Available

Installed at `.mcp-servers/mapbox/` (gitignored). Token: `MAPBOX_ACCESS_TOKEN` env var.

### Core Map & Image Generation
- `static_map_image_tool` — Generate static map PNGs (up to 1280×1280) with markers, paths, GeoJSON overlays. Style: `mapbox/dark-v11` matches our cinematic brand.
- `directions_tool` — Driving/cycling/walking routes with full GeoJSON geometry. Replaces the Remotion `buildRouteGeo` + Directions API v5 fetch.
- `search_and_geocode_tool` — Geocode airports, cities, POIs by name.
- `category_search_tool` — Search POIs by category (gas stations, hotels, motorcycle shops, etc.) with proximity bias.
- `reverse_geocode_tool` — Coordinates → address/place name resolution.
- `place_details_tool` — Rich place metadata by Mapbox ID.

### Route Intelligence
- `optimization_tool` — Multi-stop route optimization (TSP solver, 2-12 waypoints). Perfect for mission route planning.
- `matrix_tool` — Travel time/distance matrices between multiple points.
- `isochrone_tool` — Reachable area polygons by time/distance from a point.
- `map_matching_tool` — Snap GPS traces to road network (2-100 points).

### Offline Geospatial Calculations (no API calls)
- `distance_tool` — Haversine distance between coordinates.
- `bearing_tool` — Compass bearing between points.
- `midpoint_tool` — Geographic midpoint on great circle.
- `area_tool` — Polygon/multipolygon area calculation.
- `buffer_tool` — Create buffer zones around geometries.
- `bbox_tool` — Bounding box of any geometry.
- `centroid_tool` — Geometric center of polygons.
- `simplify_tool` — Douglas-Peucker geometry simplification.
- `length_tool` — LineString length measurement.
- `point_in_polygon_tool` — Point containment test.
- `points_within_polygon_tool` — Filter points inside a polygon.
- `nearest_point_tool` / `nearest_point_on_line_tool` — Proximity queries.
- `destination_tool` — Calculate destination from origin + bearing + distance.
- `convex_tool` — Convex hull of point sets.
- `union_tool` / `intersect_tool` / `difference_tool` — Boolean polygon operations.

## FFmpeg MCP Server

Installed globally via `npx -y ffmpeg-mcp-server`. Capabilities:
- Video encoding/transcoding (H.264, VP9, AV1)
- Filter graph execution (overlays, transitions, color grading)
- Dynamic metadata burn-in (route names, distances, timestamps)
- Frame extraction and image sequence assembly
- Audio mixing and subtitle embedding

## Remotion Bypass — New Pipeline

**Before (Remotion pipeline):**
```
Mission data → buildRouteGeo.ts → MapComposition.tsx (Mapbox GL JS) → Remotion render → .mp4
```

**After (Direct MCP pipeline):**
```
Mission data → mapbox:directions_tool (GeoJSON route) → mapbox:static_map_image_tool (frame sequence) → ffmpeg:assemble → .mp4
```

Key advantages:
- No headless browser / Puppeteer overhead
- No Remotion Lambda costs
- Mapbox Static Images API is ~10× faster per frame vs GL JS render
- FFmpeg handles video assembly natively with hardware acceleration
- Full pipeline controllable from agent chat via MCP tools

## Agent Usage Examples

```

Use mapbox static_map_image_tool with style dark-v11, markers at origin/destination airports, path overlay from directions_tool geometry

# Get driving directions between two airports

# Find motorcycle-friendly POIs along a route
Use mapbox category_search_tool with category "motorcycle" near route midpoint

# Assemble frame sequence into video

```
