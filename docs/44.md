# 44 — MCP + SYSTEM ARCHITECTURE CONTRACT

## 1. CORE PRINCIPLE

This system is MCP-driven.

AI MUST:
- Prefer MCP actions over scripts
- Prefer direct Firestore / Storage operations via MCP
- Avoid creating unnecessary scripts

---

## 2. DATA ARCHITECTURE

### Firestore

Primary collection:
- missionCatalogV1

Document ID:
- MUST be mission.code (deterministic)

Do NOT:
- use auto-generated IDs
- duplicate missions

---

### Storage

GPX files stored at:

missions/{slug}/route.gpx

Rules:
- Always match slug
- Always update Firestore with gpx_url after upload

---

## 3. FRONTEND ARCHITECTURE

Frontend is:

READ-ONLY for Firestore (public-safe fields)

DO NOT:
- introduce direct write logic from frontend
- expose internal fields

All writes go through:
- Firebase Functions OR MCP

---

## 4. MCP USAGE RULES

AI MUST:

- Use Firebase MCP for:
  - Firestore reads/writes
  - Storage uploads
- Use Mapbox MCP for routing
- Use Remotion MCP for video generation

DO NOT:
- simulate data
- fake responses
- invent APIs

---

## 5. FILE PATH RULES

Valid paths:

- missions/{slug}/
- functions/src/data/
- frontend/rideratlas/

DO NOT:
- invent new folders
- create duplicate structures
- change naming conventions

---

## 6. MISSION RULES

Each mission MUST have:

- code (MXP-L1, etc.)
- slug
- title
- distance_km
- route_assets:
  - gpx
  - geojson
  - polyline

Constraints:
- distance: 400–900 km
- no motorway routes

---

## 7. SAFE OPERATIONS

Before destructive actions:

AI MUST:
- ask for confirmation

Destructive includes:
- deleting collections
- overwriting missions
- bulk updates

---

## 8. NO HALLUCINATION RULE

AI MUST NOT:

- invent data
- assume file existence
- guess structure

If unsure:
- inspect or ask

---

## 9. PREFERRED WORKFLOW

1. Read existing data
2. Validate structure
3. Apply changes via MCP
4. Verify result

---

## 10. SYSTEM PRIORITY

Priority order:

1. Data integrity
2. Determinism
3. Simplicity
4. Performance

---
